| 7   | 파생 상태                | `useState` + `useEffect`로 동기화하거나, 계산으로 대체 가능한데 state로 들고 있는 경우                                     |

| 9   | Context 전에 composition | 중간 컴포넌트가 몰라도 되는 props를 알고 있나? 자기가 안 쓰고 하위 컴포넌트로 내려보낸다면 children으로 해결 가능한지 판단 |
| 10  | children vs slot         | 타입/상황별로 요소들이 달라지는데 이를 컴포넌트로 합칠 때 children을 활용한다                                              |
| 11  | Drilling vs Context      | 3단계 이상이고 composition으로 해결하기 어려운 경우 Context 도입 검토. 9번과 세트                                          |

### 판별

#### CheckoutPage

| #   | 전략         | 판정 | 판단 이유                                                                                                                                                  |
| --- | ------------ | ---- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | 변화의 경계  | ○    | 쿠폰이 바뀌어도, 배송지가 바뀌어도, 약관이 바뀌어도 이 파일을 수정해야 하는 구조. 변화의 이유가 너무 많았다                                                |
| 2   | 구현 vs 조합 | ○    | `DeliverySection` / `Form` / `AddressField`가 별도 파일이 아닌 내부 nested function으로 구현되어 있어 조합 구조가 아니었다                                 |
| 3   | God 컴포넌트 | ○    | state 9개 + 파생값 5개 + 섹션 9개. 한 컴포넌트가 알 필요 없는 것까지 전부 알고 있었다                                                                      |
| 7   | 파생 상태    | ○    | `finalPrice`가 `useState`로 마운트 시점에 고정되어 쿠폰/적립금/배송지 변경에 반응하지 않았다. `const` 파생값으로 계산해야 할 것을 state로 들고 있던 케이스 |

| #   | 전략                     | 판정 | 판단 이유                                                                                                                                                                                                                                                                                                                            |

## 과제 작업 방향

1. 제일 큰 CheckoutPage를 항목별(배송지/쿠폰/적립금 등...)로 분리했습니다. -> Section
2. Section에서 사용했던 기존 컴포넌트들을 살펴보며 도메인과 결합되어 있는 부분을 분리했습니다.
3. 도메인과 결합되어 있는 부분을 분리하며 OrderLineRow에 응집되어 있던 로직들을 각 Section으로 분리 시켰습니다.
4. UI적으로 유사한 항목들을 leaf 컴포넌트로 공통지어 common에서 관리하도록 하였습니다. (leaf 컴포넌트 - 들여쓰기 많은 태그들을 여러군데 똑같이 정의하는 것이 아닌, 재사용을 위하여 비즈니스 로직 제외한채 구현했습니다.)
5. 배송지 Section의 경우 접기|변경 을 할 수 있는 역할만 부여하고, 상세인 Form은 다른 페이지에서 사용할 수 있다는 전제 하에 분리 했습니다.
6. CheckoutPage에 있던 대량 state들을 어떤건 페이지에 남길지, 어떤건 Section으로 넘겨 Controlled, unControlled로 만들어야 할지 기준을 정하여 수정했습니다.

## 사용성 개선

1. 쿠폰, 적립금 사용 순서에 따라 마이너스 금액이 노출되는 현상이 있어 수정했습니다.
2. 주문 완료 후 다시 주문/결제 페이지로 이동했을 때 이전 주문 데이터 초기화 했습니다.
3. 금액 입력란(적립금)의 default 값이 0인데 입력할 때 05, 050으로 입력되어 이를 5, 50으로 변환되도록 하였습니다.
